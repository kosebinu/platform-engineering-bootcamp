# Kubernetes Monitoring Stack Architecture

## Purpose

This document describes the monitoring platform used by the Platform Engineering Team to observe, monitor, and operate Kubernetes workloads.

Modern Kubernetes platforms consist of hundreds or thousands of applications running across multiple clusters. A complete monitoring solution must provide visibility into application performance, infrastructure health, Kubernetes resources, and platform reliability.

Our platform uses the **kube-prometheus-stack** Helm chart to deploy a production-ready monitoring solution that includes Prometheus, Grafana, Alertmanager, kube-state-metrics, Node Exporter, and the Prometheus Operator.

# Prometheus

Prometheus is the central metrics collection system.

Its responsibilities include:

- Discovering monitoring targets
- Scraping metrics from applications
- Storing time-series data
- Executing PromQL queries
- Evaluating alert rules

Applications expose metrics through a `/metrics` endpoint, which Prometheus periodically scrapes.

Example workflow:

```
Application

↓

/metrics

↓

Prometheus

↓

Time-Series Database
```

Prometheus uses a **pull-based** architecture, periodically collecting metrics rather than requiring applications to push data.

---

# Grafana

Grafana provides visualization and dashboards for collected metrics.

Platform Engineers use Grafana to:

- Monitor Kubernetes clusters
- Visualize application performance
- Track infrastructure utilization
- Analyze production incidents
- Build operational dashboards

Grafana queries data sources such as Prometheus rather than storing metrics itself.

Typical dashboards include:

- Cluster Overview
- Node Health
- Namespace Utilization
- Application Performance
- Deployment Health
- Resource Consumption

Grafana serves as the primary user interface for platform observability.

---

# Alertmanager

Alertmanager is responsible for processing alerts generated by Prometheus.

Its responsibilities include:

- Routing alerts
- Deduplicating repeated alerts
- Grouping related alerts
- Silencing maintenance alerts
- Sending notifications

Alert destinations may include:

- Email
- Slack
- Microsoft Teams
- PagerDuty
- Webhooks

Example workflow:

```
Prometheus

↓

Alertmanager

↓

Slack

PagerDuty

Email
```

Prometheus detects problems.

Alertmanager notifies engineers.

---

# kube-state-metrics

kube-state-metrics exposes information about Kubernetes objects as Prometheus metrics.

Unlike Node Exporter, kube-state-metrics does not monitor hardware.

Instead, it monitors Kubernetes resources such as:

- Deployments
- ReplicaSets
- Pods
- StatefulSets
- DaemonSets
- Jobs
- CronJobs
- Nodes
- Namespaces
- Horizontal Pod Autoscalers

Example:

```
Kubernetes API

↓

kube-state-metrics

↓

Prometheus
```

This enables dashboards showing deployment health, replica counts, pod status, and workload readiness.

---

# Node Exporter

Node Exporter exposes operating system metrics from Kubernetes worker nodes.

It provides metrics such as:

- CPU utilization
- Memory utilization
- Filesystem usage
- Disk I/O
- Network traffic
- Load average
- System uptime

Example:

```
Linux Node

↓

Node Exporter

↓

Prometheus
```

These metrics allow Platform Engineers to monitor cluster infrastructure rather than just applications.

---

# Prometheus Operator

The Prometheus Operator simplifies the management of Prometheus within Kubernetes.

Instead of manually editing Prometheus configuration files, engineers define Kubernetes custom resources such as:

- ServiceMonitor
- PodMonitor
- PrometheusRule
- AlertmanagerConfig

The Operator continuously watches these resources and automatically updates Prometheus configuration.

This follows Kubernetes' declarative model and eliminates manual configuration management.

---

# ServiceMonitor

A ServiceMonitor tells Prometheus how to monitor an application.

It defines:

- Which Kubernetes Service to monitor
- Which endpoint to scrape
- Which port to use
- How frequently to scrape metrics

Example:

```
Application

↓

Service

↓

ServiceMonitor

↓

Prometheus
```

Application teams expose metrics, while the Platform Team standardizes monitoring through ServiceMonitors.

# Why Helm is Used

Deploying a production monitoring platform manually requires managing dozens of Kubernetes resources.

Examples include:

- Deployments
- Services
- StatefulSets
- ConfigMaps
- Secrets
- CRDs
- ServiceAccounts
- Roles
- RoleBindings

Helm packages these resources into a reusable deployment.

Benefits include:

- Simplified installation
- Consistent upgrades
- Version management
- Easy rollback
- Configuration through values files
- Standardized deployments

Helm enables Platform Teams to treat monitoring as a reusable platform product.

# kube-prometheus-stack

Our platform deploys the **kube-prometheus-stack** Helm chart.

This chart is maintained by the Prometheus Community and provides a complete monitoring solution.

It includes:

- Prometheus
- Grafana
- Alertmanager
- Prometheus Operator
- kube-state-metrics
- Node Exporter
- Default dashboards
- ServiceMonitors
- PrometheusRule CRDs

Using this Helm chart significantly reduces operational complexity while following Kubernetes best practices.

# Platform Engineering Principles

Our monitoring platform follows these principles:

- Monitoring is provided as a platform capability.
- Every application exposes metrics.
- Infrastructure monitoring is centralized.
- Dashboards are reusable and standardized.
- Alerting is automated and declarative.
- Monitoring configuration is managed as code.
- Platform components are deployed using Helm.
- Observability integrates with GitOps and Progressive Delivery.

# Key Takeaways

- The monitoring stack consists of multiple specialized components working together.
- Prometheus collects and stores time-series metrics.
- Grafana provides dashboards and visualization.
