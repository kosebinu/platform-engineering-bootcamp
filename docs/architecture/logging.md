# Centralized Logging Architecture

## Purpose

This document describes the centralized logging architecture used by the Platform Engineering Team to collect, store, search, and analyze logs generated by Kubernetes workloads.

Applications running on Kubernetes are ephemeral by design. Pods may be rescheduled, restarted, or deleted at any time, making local log storage unreliable. To ensure logs remain available for troubleshooting, auditing, and incident response, the platform aggregates logs into a centralized logging system.

Our platform uses **Grafana Loki** for log storage and **Promtail** as the log collection agent. Grafana provides a unified interface for searching and correlating logs with metrics and traces.

---

# Why Centralized Logging is Needed

Traditional infrastructure often stored logs locally on application servers.

Example:

```
Server

↓

/var/log/application.log
```

Engineers connected to the server using SSH to investigate issues.

This approach does not work well in Kubernetes because Pods are temporary.

Example:

```
Payment API Pod

↓

Crash

↓

Pod Deleted

↓

Local Logs Lost
```

Without centralized logging:

- Logs disappear when Pods are deleted.
- Engineers must manually inspect multiple Pods.
- Troubleshooting becomes slow and inconsistent.
- Cross-cluster log analysis is nearly impossible.

Centralized logging ensures that logs remain available even after workloads terminate.

# Kubernetes Logging Model

Kubernetes recommends that applications write logs to standard output (`stdout`) and standard error (`stderr`) rather than writing directly to files inside the container.

Example:

```
Application

↓

stdout / stderr

↓

Kubelet

↓

Container Runtime

↓

Log Collector

↓

Central Logging Platform
```

This approach allows Kubernetes to manage log collection consistently across all workloads.

Applications should avoid storing logs inside container filesystems because containers are immutable and short-lived.

---

# stdout and stderr

Applications should use the following streams:

## stdout

Standard output is used for normal application events.

Examples:

- Application startup
- Request processing
- Informational messages
- Successful operations

Example:

```
INFO Payment processing started
```

---

## stderr

Standard error is used for unexpected conditions.

Examples:

- Exceptions
- Failed API calls
- Database errors
- Validation failures

Example:

```
ERROR Payment provider timeout
```

By separating normal output from error output, logging systems can process events more effectively.

# Loki Architecture

Grafana Loki is the centralized log storage system used by the platform.

Unlike traditional log platforms, Loki indexes metadata (labels) rather than the full log contents.

This reduces storage costs while maintaining efficient search capabilities.

High-level workflow:

```
Application

↓

stdout / stderr

↓

Promtail

↓

Loki

↓

Grafana
```

Loki provides:

- Centralized log storage
- Label-based indexing
- Fast log search
- Grafana integration
- Multi-tenant support
- Kubernetes-native architecture

---

# Promtail Responsibilities

Promtail is the log collection agent responsible for delivering logs to Loki.

Promtail performs the following tasks:

- Watches Kubernetes Pod logs
- Reads stdout and stderr streams
- Adds Kubernetes metadata
- Applies log processing pipelines
- Sends logs to Loki

Example:

```
Pod Logs

↓

Promtail

↓

Namespace Labels

↓

Pod Labels

↓

Loki
```

Although newer agents such as Grafana Alloy and the OpenTelemetry Collector are increasingly common, Promtail remains useful for understanding the fundamentals of Loki-based log collection.

# Structured Logging

Applications should generate structured logs whenever possible.

Avoid free-form log messages.

Poor example:

```
Something failed
```

Recommended JSON format:

```json
{
  "timestamp": "2026-07-13T14:25:16Z",
  "level": "ERROR",
  "service": "payment-api",
  "namespace": "payment",
  "orderId": "O12345",
  "customerId": "C67890",
  "durationMs": 4821,
  "message": "Payment provider timeout"
}
```

Structured logging enables:

- Easier searching
- Better filtering
- Automated parsing
- Log analytics
- Correlation with metrics and traces

---

# Log Levels

Applications should use standardized log levels.

| Level | Purpose |
|--------|----------|
| DEBUG | Detailed diagnostic information for development and troubleshooting |
| INFO | Normal application events |
| WARN | Recoverable or unexpected situations |
| ERROR | Failed operations requiring attention |
| FATAL | Critical failures causing application termination |

Consistent log levels improve alerting, filtering, and operational visibility across the platform.

---

# Labels

Loki indexes labels instead of the complete log contents.

Typical labels include:

```
cluster = aks-prod

namespace = payment

pod = payment-api-6df8c

container = payment-api

environment = production
```

Labels allow engineers to quickly narrow log searches.

Examples:

Search all logs from the payment namespace:

```
namespace="payment"
```

Search all logs from production:

```
environment="production"
```

Search all logs for a specific Pod:

```
pod="payment-api-6df8c"
```

Using labels efficiently improves query performance and simplifies troubleshooting.

# Grafana Integration

Grafana provides the primary interface for exploring logs stored in Loki.

Engineers can:

- Search logs
- Filter by labels
- View log streams
- Correlate logs with metrics
- Investigate production incidents
- Navigate between dashboards and log views

Typical workflow:

```
Grafana Dashboard

↓

High CPU Alert

↓

Open Logs

↓

Find Error

↓

Identify Root Cause
```

Grafana enables a seamless investigation experience by combining metrics, logs, and traces within a single interface.

# Platform Engineering Principles

Our centralized logging platform follows these principles:

- Applications write logs to stdout and stderr.
- Logs are collected automatically by the platform.
- Structured logging is the organizational standard.
- Labels are used for efficient searching.
- Logging infrastructure is centrally managed.
- Logs are correlated with metrics and traces.
- Logging configuration is managed as code.
- Platform teams provide logging as a self-service capability.

---

# Key Takeaways

- Kubernetes workloads are ephemeral, making centralized logging essential.
- Applications should write logs to stdout and stderr instead of local files.
- Promtail collects logs from Kubernetes workloads and forwards them to Loki.
- Loki stores logs efficiently using label-based indexing.
- Structured logging improves searchability and operational analysis.
- Standardized log levels provide consistency across applications.
- Grafana provides a unified interface for log exploration and troubleshooting.
- Centralized logging is a foundational capability of a modern Platform Engineering platform and complements metrics and distributed tracing to deliver complete observability.
